Pi Analytics Desktop

Readability and Maintainability Improvement Report

The desktop app has a promising product shape: an Electron shell, a typed IPC boundary, an agent-driven canvas, generated HTML artifacts, parallel task cards, and persisted boards. The main readability issue is not the lack of React or Tailwind. It is that the renderer currently concentrates too many responsibilities into one large imperative module.

Recommendation: extract first, migrate later Reviewed package: packages/analytics-desktop Report date: 2026-06-30

Executive Summary

  • Do not start with a full React migration. It would add a second large change while the current renderer boundaries are still unclear.
  • Use Svelte only after a short extraction phase. It is the best framework fit if the goal is less UI boilerplate and more readable component files.
  • Do not adopt Tailwind as the main readability fix. The current semantic CSS is easier to scan than long utility class strings for this app.
  • First priority: split renderer state, canvas math, card rendering, chat rendering, sanitization, and event handling into focused modules with pure functions.
  • Second priority: add tests around extracted pure logic before any framework change. This reduces migration risk and catches interaction regressions.

Why The Renderer Feels Hard To Read

Observed Shape Readability Impact Concrete Improvement
renderer/src/main.ts owns app state, DOM lookup, rendering, event handling, canvas math, persistence triggers, markdown parsing, and sanitization. A reader must hold the entire app in memory to safely change one interaction. Extract modules by responsibility before adding features.
Rendering is imperative and broad. render() updates auth, status, models, chat, queue, cards, selection, viewport, and canvas paint scheduling. Small state changes can cause large redraw paths. It is hard to see which function owns which DOM subtree. Split into view-specific render functions behind a narrow state/view-model boundary.
Canvas cards are rendered twice: DOM cards in cardHost and cloned card nodes under workspaceCanvas for HTML-in-canvas painting. Duplicate rendering increases the chance of drift and makes canvas bugs harder to reason about. Centralize card element creation and isolate HTML-in-canvas fallback behavior.
Global mutable state drives selection, tasks, busy status, modal state, loading cards, viewport, and dock preferences. Call order matters. Behavior is correct only if every handler remembers to call the right render and save functions. Introduce explicit actions and pure reducers for state transitions.
Markdown parsing and HTML sanitization live inside the renderer entrypoint. Security-sensitive and formatting-sensitive code is mixed with UI event wiring. Move to dedicated modules and test dangerous inputs.

Proposed Renderer Module Split

The goal is to make a reader understand one concept per file. This can be done before any framework migration and should keep the public behavior unchanged. 

packages/analytics-desktop/renderer/src/
  main.ts                         # bootstraps app, wires top-level listeners
  app-state.ts                    # AppState, initialState, actions, reducer
  dom.ts                          # requireElement and DOM references
  preferences.ts                  # localStorage-backed dock preferences
  cards/
    render-card.ts                # createCardElement, appendCardBody
    markdown.ts                   # appendMarkdown, inline markdown
    sanitize-canvas-html.ts       # sanitizeCanvasHtml and CSS sanitizer
  canvas/
    viewport.ts                   # pan, zoom, coordinate math
    placement.ts                  # resolveNewCardPosition and overlap helpers
    paint.ts                      # grid paint and HTML-in-canvas behavior
  chat/
    render-chat.ts                # chat messages, streaming, queue
    tasks.ts                      # TaskView helpers and active task model
  commands/
    board.ts                      # save board, new board, export
    selection.ts                  # select, keep, delete, ask selected
Key rule Do not move code by category alone. Move it around stable behavior seams: state transition, render subtree, canvas math, IPC command, or parsing/sanitization.

Target Architecture

Readability Work Plan

Week 1

Extract pure logic without behavior changes

Move card placement, viewport math, markdown parsing, HTML sanitization, compact path formatting, task helpers, and model value helpers into focused modules. Add unit tests for each extracted module.

Week 2

Separate state transitions from rendering

Introduce action functions or a reducer for auth, session snapshots, renderer events, selection, card updates, task updates, and dock preferences. Keep the DOM implementation plain TypeScript.

Week 3

Split view modules and reduce redraw scope

Give each DOM subtree an owner: topbar, login modal, chat dock, task tabs, queue, canvas card host, context bar, focus modal, and toast. Replace broad rerenders where small targeted updates are clearer.

Week 4

Decide on Svelte pilot

If the extracted renderer is still too verbose, pilot Svelte on one isolated surface such as the chat dock. Do not migrate canvas math, agent IPC, or sanitizer code into component files.

Tailwind Assessment

Question Answer For This App
Would Tailwind make styling faster? Probably, for new layout work and repetitive spacing. It would not address the main readability problem in main.ts.
Would Tailwind make code easier to read? Not by default. The existing CSS uses semantic classes like canvas-card, chat-dock, and artifact-panel, which are readable in a domain-specific UI.
Where would Tailwind hurt? Generated artifact HTML, long component markup, and dense interaction surfaces where class strings can obscure structure.
Best compromise Keep semantic CSS variables and named classes. Add small utility classes or design tokens only if duplication becomes measurable.

Testing Plan For Readability Refactors

High-value unit tests

  • resolveNewCardPosition, overlap handling, and placement candidates.
  • Viewport zoom and pan math, especially pointer-centered zoom.
  • Selection commands: additive selection, delete, keep, active task behavior.
  • Markdown inline parsing and list/heading/code block handling.
  • Canvas HTML sanitizer for script tags, event handlers, javascript URLs, and CSS URLs.
  • Renderer event application for session snapshots, task updates, streams, and queue updates.

Why this matters

The existing package already has focused tests around card events, task dispatch, task controller behavior, and canvas tool behavior. Extending that pattern to renderer logic will make future framework migration lower risk because behavior can be preserved while the view layer changes.

Testing constraint Prefer Node/Vitest tests for extracted pure modules. Avoid Electron UI tests until pure logic coverage exists.

Risks And Dependencies

Risk Why It Matters Mitigation
High Big-bang framework migration Large behavior and architecture change at the same time. Hard to review and easy to regress canvas interactions. Extract first. Migrate one surface only after tests exist.
Medium HTML-in-canvas experimental behavior The app enables experimental CanvasDrawElement features and keeps DOM fallback behavior. This is inherently special-case code. Isolate paint logic and fallback rules in one module with clear comments and tests for state toggles.
Medium Generated HTML artifact safety Agent-generated HTML is rendered into the DOM after custom sanitization. Move sanitizer to a dedicated module, add malicious fixture tests, and keep scripts disabled.
Medium CSS growth The stylesheet is large but still semantically organized. Random utility adoption could make it harder to inspect. Split CSS by surface or use Svelte scoped styles if a framework is adopted.
Low Main process shape Main process code is comparatively focused: Electron window, IPC validation, board persistence, report export, and agent controller delegation. Keep this boundary. Improve export report rendering separately if needed.
